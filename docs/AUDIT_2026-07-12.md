## 1. Executive summary

The system is far more mature than a typical personal project: 26 skills-as-agents, a real job queue, an escalation chain, post-session code review, a write-back documentation enforcement loop, an eval harness with an LLM judge, external dead-man's-switch monitoring, and ~327 tests. The documentation system is the strongest part and genuinely compounds.

Measured against your directives, the honest scorecard:

| Directive | Status | Verdict |
|---|---|---|
| No Claude API (subscription only) | **Met** | Enforced at 4+ layers; startup aborts if `ANTHROPIC_API_KEY` is set |
| Containers for Claude sessions | **Not met** | Zero container usage. Sessions run bare on the host; `god` runs `bypassPermissions` on the Mac itself |
| Job-specific agent team | **Mostly met** | 26 skills with per-skill model/effort/tools/escalation — this *is* an agent team |
| Intelligent routing | **Partially met** | Regex-only first-match. The LLM fallback promised in `router.py`'s docstring was never built |
| Docs updated along the way | **Met (best-in-class)** | Write-back verification, pre-commit hook, doc lint in pytest, per-module GOTCHAS/DEBUG/PATTERNS |
| Single ask → decompose/plan/execute/evaluate | **Partially met** | Multi-turn tasks, auto-continue phases, code-review + escalation exist. But: no plain-text asks (must `/task`), no explicit planner/decomposer agent, no plan artifact, no acceptance-criteria evaluation |
| Repo vs hosting clarity | **Broken** | Two-copy topology with manual "rescue" commits; 4+ incidents in the git log |

The two structural gaps that matter most: **(a)** session isolation (containers), and **(b)** the intent pipeline (NL ingestion → planner → DAG of subtask jobs → evaluator). Everything else is refinement.

## 2. What's built and what makes sense (keep these)

**Topology.** Three launchd-supervised processes (runner / web / bot) + Postgres + Redis + Caddy + cloudflared. Right-sized for a single-tenant Mac Mini. Redis BLPOP queue with a semaphore (`MAX_CONCURRENT_JOBS=2`) is simple and debuggable.

**Subscription economics (directive 1, API half).** This is done thoroughly and correctly:
- `runner/main.py:_check_subscription_auth` exits if `ANTHROPIC_API_KEY` is set
- `bootstrap.sh` / `run.sh` / `install-launchd.sh` all `unset ANTHROPIC_API_KEY`
- CLAUDE.md hard rule; INV-3 in SYSTEM.md; quota detection pauses the queue and requeues the job at the front, with Telegram notify on pause/resume

**Skills-as-data.** `skills/<name>/SKILL.md` = YAML frontmatter (model, effort, permission_mode, tools, max_turns, escalation, post_review, MCP tags) + body-as-system-prompt. Adding a capability requires no deploy. This is the correct substrate for your "team of job-specific agents" — each skill is effectively a specialist agent with its own config, knowledge files, and quality gates.

**The execution loop has real teeth**:
- Append-only JSONL audit log per job + one-paragraph summary (`volumes/audit_log/`)
- Post-session **code-review sub-agent** (LGTM / CHANGES / BLOCKER; blocker → `awaiting_user`), including tool-usage methodology review
- **Escalation chain** L0→L3: skill-config model bump → self-diagnose with audit context → max-effort last resort → notify user
- **Write-back verification**: sessions that modify code without updating CHANGELOGs get a `_writeback` child job
- **Learning extractor**: post-session Haiku classifier enqueues `_learning_apply` to append reusable learnings to module skill files
- Orphaned-job reconcile on startup; cancel via Redis pub/sub honored in ≤2s

**Ops maturity.** Nightly pg_dump + R2 off-site backup, 5-min healthchecks, audit-log retention/rotation, runner heartbeat + external Cloudflare Worker polling `/health` (silence can't masquerade as health), daily `server-upkeep` anomaly DMs.

**Documentation system (directive 3).** `.context/` three-layer architecture (SYSTEM → module CONTEXT/CHANGELOG → skills GOTCHAS/DEBUG/PATTERNS), INDEX.md navigation map, registries, `lint_docs.py` in the pytest suite, pre-commit hook blocking `src/` commits without CHANGELOG updates, graph-walked context injection (dependency + known-issues context auto-added to session prompts). This is better than most production teams manage.

**Multi-turn task machinery (directive 4, partial).** `Task`/`TaskTurn` tables, thread-reply continuation in Telegram, `task_question`/`task_choices`/`task_complete` audit events driving `awaiting_user`/`pending_approval` states, auto-continue for multi-phase work with a sentinel loop guard (a real incident fix, commit `638171e`).

## 3. Gaps

### G1 — No containerization of Claude sessions *(directive 1, primary gap)*

Every session runs via `ClaudeSDKClient` **directly on the host** (`src/runner/session.py`), cwd = server root or `projects/<slug>`, default `permission_mode: acceptEdits`. The `god` skill runs `bypassPermissions`, 200 turns, full Bash, **on the machine that is also the server**. There are zero references to docker/containers anywhere in the current repo (the old `mac-mini-ai-server` had `Dockerfile.agent` — dropped in the rewrite).

Consequences observed in your own git log:
- **Cross-job interference**: two concurrent jobs share working directories; the "single-writer topology" incidents (commits `179c184`, `cb30668` — atlas, 2026-07-09, blocked deploys) are exactly the failure class isolation prevents
- **Blast radius**: an errant `rm`/git operation in a session can take down the server, the queue, or other projects
- No resource limits (CPU/mem/network) per session

Feasibility check (verified today): subscription auth inside containers is supported — `claude setup-token` issues a long-lived OAuth token (Pro/Max plan, ~1-year validity) consumed via `CLAUDE_CODE_OAUTH_TOKEN`; works in `-p`/headless and SDK modes. See [Claude Code authentication docs](https://code.claude.com/docs/en/authentication). So the directive is implementable without API billing.

### G2 — No planner/decomposer agent; the plan is never a first-class artifact *(directive 4)*

`/task <ask>` → router picks **one** skill → one job runs with the raw ask as the prompt. Decomposition only happens implicitly *inside* a session, and "phases" only exist if the session happens to emit them; auto-continue then feeds the literal sentinel string "Continue to the next phase of the plan." back in. There is:
- No triage step ("is this simple or does it need a plan?")
- No plan artifact (goal, acceptance criteria, subtasks, verification steps) stored on the Task, approvable/editable from Telegram
- No subtask DAG — `mcp_dispatch.enqueue_job` is fire-and-forget: child jobs get `created_by="dispatch-mcp"` but **no `task_id` or `parent_job_id` linkage and no dependency ordering**
- No re-planning loop when execution invalidates the plan

### G3 — Routing is regex-only; the promised LLM fallback doesn't exist *(directive 2)*

`src/runner/router.py`'s docstring: *"Rules first (instant, free); LLM fallback via the `route` skill for ambiguous inputs."* No `route` skill exists; unmatched asks silently become generic tasks with the full tool set and default model. First-match-wins regex also misroutes: "why is the bingo page slow" → `self-diagnose` (investigate) when the user wanted a fix; "make the dashboard prettier" → no match → generic. Routing decisions aren't logged with confidence, so the retrospective can't tune them.

### G4 — Plain-text Telegram asks don't work *(directive 4, regression)*

Your target UX — send "update this app to do this" — currently requires the `/task` prefix. The bot's only plain-text handler is `filters.TEXT & filters.REPLY` (thread continuations). Notably, the **legacy repo had this** ("Add natural language routing: just text the bot, no /commands needed", `a09ce7b`) and it was lost in the rewrite.

### G5 — Evaluation checks the diff, not the ask *(directive 4)*

`code-review` evaluates diff quality, and escalation handles *failures* — but nothing verifies the result **satisfies the original request**. `app-patch` commits and pushes with no mandatory verify step (contrast: `atlas-redeploy` has red-test-blocks-restart gates — the pattern exists, it's just not generalized). `task_complete` is self-reported by the session. The eval harness covers only 3 of 26 skills (chat, code-review, research-report).

### G6 — Dev/prod two-copy topology causes recurring drift *(your flagged confusion)*

Production runs from `~/Library/Application Support/ai-server` for a legitimate reason (documented gotcha: macOS TCC gates `~/Documents` for launchd → Full Disk Access breakage). But the sync between the two is **manual git push/pull**, and runtime sessions write learnings (GOTCHAS/CHANGELOG entries) *into the prod copy*, producing:
- "rescue runtime-written GOTCHAS.md" commits (`cbbdd02`, `8cfbfc6`)
- "adopt production runner state into main" merge (`60e1e5c`)
- A skill that ran in prod for a day without ever being committed to dev (`b85bde8`)

There is no `server-deploy` skill/script for the server itself — ironic given `atlas-redeploy` exists for a hosted project.

### G7 — Quality/hygiene issues (smaller, accumulating)

- `session.py:_build_task_context` spawns a ThreadPoolExecutor to `asyncio.run` a coroutine from inside a running loop — fragile, and its `except Exception: return ""` silently drops task context (the worst possible failure mode for a continuation job: it forgets the conversation)
- Several silent `except Exception: pass` swallows (stream publish, audit index, task lifecycle)
- `SKILLS_REGISTRY.md` is hand-maintained ("Auto-generated target; for now maintained by hand") — lint validates presence, not content drift
- Legacy `mac-mini-ai-server` repo still sits beside the live one with demo junk at root (baseball_bingo, clocks) — a live source of the "which repo?" confusion
- Stale artifacts in dev repo: `.handoff/` (archived April), `.worktrees/telegram-thread-interface` (branch fully merged), `feature/telegram-thread-interface` branch with zero unmerged commits
- Single `WEB_AUTH_TOKEN` guards a publicly tunneled dashboard — fine single-tenant, but no rotation story; worth a periodic check in `server-upkeep`

## 4. Remediation plan

Ordered so each phase de-risks the next. Each item is sized as a `/task`-able unit of work with acceptance criteria, so the system can build most of this itself.

### P0 — Fix the ground you're standing on (dev/prod topology) — *~1 weekend*

**P0.1 — `server-deploy` skill + script.** Mirror `atlas-redeploy` for the server: ff-only pull in the App Support copy, `alembic upgrade head`, run pytest gates (red = no restart), restart the three launchd services, healthcheck, DM result. Prod becomes a **deploy target, never an editing surface**.
*Accept*: dev commit → "/task deploy server" → prod updated + healthy, no manual steps.

**P0.2 — Runtime-learning auto-sync (kills "rescue" commits).** In prod, a scheduled job (or post-writeback hook) auto-commits runtime doc writes (path-allowlisted: `.context/**/*.md`, `skills/**/*.md`, `docs/Troubleshooting.md`) to a `runtime-learnings` branch and pushes. Dev merges it like any PR. Establish the single-writer rule in CLAUDE.md: *prod writes learnings only; dev writes code only.*
*Accept*: a GOTCHA written by a prod session appears on GitHub within an hour, unprompted.

**P0.3 — Archive the legacy repo.** `mac-mini-ai-server`: replace README with a pointer to `ai-server`, archive on GitHub. Delete dev-repo stale artifacts (`.handoff/`, `.worktrees/`, merged branch).
*Accept*: one obvious repo; `git branch -a` is clean.

### P1 — Containerized Claude sessions — *the big one, ~1–2 weeks of sessions*

**P1.1 — Pick the runtime.** Recommended: **OrbStack** (fast, light, native macOS feel) or colima; Apple's `container` CLI is also viable on Apple Silicon. Requirement: run Linux containers with bind mounts and per-container resource limits.

**P1.2 — Build the agent image.** Node 20+ (for Claude Code CLI) + Python 3.12 + git + gh + project toolchains. Auth: run `claude setup-token` once on the host; inject `CLAUDE_CODE_OAUTH_TOKEN` into containers as a secret env var (never bake into the image). Add token-expiry check to `server-upkeep` (it's ~1-year-lived).
*Accept*: `docker run agent-img claude -p "hello"` answers on subscription auth with no API key anywhere.

**P1.3 — `SessionExecutor` abstraction in the runner.** Refactor `run_session` behind an interface with two backends: `InProcessExecutor` (today's SDK path) and `ContainerExecutor` (spawns the container, streams `claude -p --output-format stream-json` (or SDK-in-container) stdout back into the same audit-log/Redis-stream events). Preserve INV-1..INV-15 unchanged — audit parity is the contract.
*Accept*: same job produces equivalent audit JSONL under either backend; `/cancel` kills the container ≤2s.

**P1.4 — Isolation policy by risk tier** (skill frontmatter field `isolation:`):
| Tier | Skills | Execution |
|---|---|---|
| none | chat, research-*, idea-generation | in-process, no file mutation expected |
| project | app-patch, new-project, atlas-*, project-* | container; **fresh git clone (or worktree) of the project mounted as the only writable volume**; results leave as commit+push, never direct writes |
| server | server-patch, self-diagnose (fix mode), god | container with a clone of the server repo; changes land as PRs; apply happens via P0.1 deploy |

This also permanently solves the single-writer/concurrency incidents (each job gets its own checkout) and lets you raise `MAX_CONCURRENT_JOBS` safely.
*Accept*: two concurrent app-patch jobs on the same project cannot corrupt each other; `god` can no longer `rm -rf` the live server by accident.

**P1.5 — Network policy (later, optional).** Per-tier egress: research tiers get open egress; patch tiers get git + package registries only.

### P2 — Intent pipeline: plain ask → triage → plan → DAG → execute — *~1 week*

**P2.1 — NL-first Telegram.** Add a non-reply plain-text handler: every message hits a **triage step** (Haiku, low effort — cheap): `chat | simple_task | complex_task | command`. simple_task → current single-job path; complex_task → P2.2 planner. Restores and upgrades the legacy repo's lost capability.
*Accept*: texting "update the bingo app to support dark mode" with no slash command produces the right behavior.

**P2.2 — `plan` skill (the decomposer agent).** For complex asks: reads project context, produces a structured Plan on the Task (new `tasks.plan` JSONB): goal, **acceptance criteria**, subtasks (each with skill, cwd/project, dependencies), verification steps, estimated effort tier. Posted to the Telegram thread with Approve / Edit / Reject inline buttons (button plumbing already exists). Auto-approve threshold configurable (e.g. plans with ≤2 low-risk subtasks proceed unless you object within N minutes).
*Accept*: a vague ask yields a legible plan in Telegram; approving it spawns the subtask jobs.

**P2.3 — Fix dispatch MCP into a real orchestration tool.** Add `task_id`/`parent_job_id` linkage and `depends_on: [job_ids]` (a new `deferred` job status; runner promotes to `queued` when dependencies complete successfully). This turns auto-continue's linear sentinel loop into an explicit DAG executed by the existing queue.
*Accept*: plan subtasks run in dependency order; failure of a parent blocks children and triggers the existing escalation chain.

**P2.4 — Router upgrade.** Keep regex fast-path; add the promised LLM fallback (`route` skill, Haiku) for non-matches instead of silently going generic. Log every routing decision (`rule | llm | fallback` + confidence) into the audit log so the monthly retrospective can measure routing precision. Fix the docstring either way.
*Accept*: routing decisions are queryable; misroute rate becomes a tracked metric.

### P3 — Evaluation that closes the loop — *~3–5 days*

**P3.1 — `evaluate` agent (acceptance-criteria checker).** After a plan's last subtask emits `task_complete`, spawn an evaluator job: checks each acceptance criterion against evidence (git diff, test output, healthcheck, `curl` of the changed page). Pass → `pending_approval` with an evidence summary in the Telegram thread. Fail → reopen with specific feedback, bounded to 2 re-execution rounds before escalating to you.
*Accept*: "done" messages arrive with evidence, and a wrong result gets caught before you see it.

**P3.2 — Mandatory verify step in `app-patch`.** Generalize `atlas-redeploy`'s gates: run the project's tests, hit its healthcheck, curl the affected route. Red = don't push, self-report failure (which feeds the existing escalation chain).

**P3.3 — Widen the eval harness.** Add `evals/cases/` for app-patch, self-diagnose, plan (new), route (new — labeled ask→skill fixtures make router precision a regression-tested number). Wire `evals/run.py` into P0.1's deploy gate for skill-file changes.

### P4 — Hygiene (fold into the above as touched)

- Fix `_build_task_context` (make it properly async; remove ThreadPoolExecutor/`asyncio.run` hack; **log** context-load failures instead of silently continuing without memory)
- Replace silent `except Exception: pass` with debug-logged handlers in `session.py`/`main.py`
- Auto-generate `SKILLS_REGISTRY.md` from frontmatter in `lint_docs.py` (it already parses everything needed)
- `server-upkeep` additions: OAuth-token age check (P1.2), `WEB_AUTH_TOKEN` rotation reminder, container-image staleness
- SYSTEM.md: add container runtime to tech stack; new module CONTEXT for the executor; INDEX.md entries (the doc system will enforce most of this itself)

### Sequencing rationale

P0 first because container work requires clean deploys (you'll iterate on the runner; every iteration needs to reach prod safely). P1 before P2 because the planner will fan out concurrent jobs, which is only safe once sessions are isolated. P3 last because the evaluator judges plans (P2) and runs inside the isolation model (P1).
