# Idea Generation

You generate novel, actionable ideas on a given topic and write them to a dated
markdown file in `projects/ideas/`. All ideas are deduped against prior ideas
stored in `projects/ideas/history.jsonl`. Ideas are **private** -- this project
is never publicly served.

## Inputs you will receive

Extract from the job description (and optionally `payload`):
- **topic** (required): what to brainstorm about
- **count** (default=3): how many final ideas to produce (range: 1-5)
- **style** (default=actionable): one of `actionable | speculative | technical-only | product-only`
  - `actionable`: ideas you could start executing this week
  - `speculative`: wild, high-upside ideas that may not be feasible yet
  - `technical-only`: engineering/architecture ideas only, no business/product
  - `product-only`: product/UX/market ideas only, no raw engineering

## Procedure

1. **Ensure `projects/ideas/` exists and is a git repo.** On first run, the
   project directory may not exist yet. Bootstrap it:
   ```bash
   if [ ! -d projects/ideas/.git ]; then
     mkdir -p projects/ideas/.context
     touch projects/ideas/history.jsonl
     cat > projects/ideas/CLAUDE.md << 'SCAFF'
   # Ideas

   Private storage for ideas generated by the `idea-generation` skill.

   ## Key rules
   - Ideas are PRIVATE -- not served publicly, no manifest.yml
   - Each batch is a dated markdown file: `<topic-slug>-YYYY-MM-DD.md`
   - All ideas also appended to `history.jsonl` for dedup
   - Read `.context/CONTEXT.md` for conventions
   SCAFF
     cat > projects/ideas/.context/CONTEXT.md << 'SCAFF'
   # Ideas project

   ## Purpose
   Private storage for ideas generated by the `idea-generation` skill.

   ## Structure
   ```
   projects/ideas/
   +-- .git/
   +-- .context/
   |   +-- CONTEXT.md
   |   +-- CHANGELOG.md
   +-- CLAUDE.md
   +-- history.jsonl
   +-- <topic-slug>-YYYY-MM-DD.md  (idea batches)
   ```

   ## Not hosted
   No manifest.yml. Not exposed via Caddy. Private content storage.
   Ideas are never served publicly.

   ## Dedup
   `history.jsonl` contains one JSON line per idea ever generated.
   New ideas are checked against this file to avoid repeats.
   SCAFF
     cat > projects/ideas/.context/CHANGELOG.md << 'SCAFF'
   # Ideas catalog

   <!--
   Every idea-generation run appends an entry at the top.
   Newest entries above. Format:

   ## YYYY-MM-DD -- <topic>
   File: <slug>-YYYY-MM-DD.md
   Count: N ideas
   Style: actionable | speculative | technical-only | product-only

   -->
   SCAFF
     ( cd projects/ideas && git init -q && git add -A && \
       git commit -q -m "Initial scaffold from idea-generation skill" )
   fi
   ```

2. **Load prior ideas for dedup.** Read `projects/ideas/history.jsonl`. Each
   line is a JSON object with at least `{ "title": "...", "topic": "...",
   "date": "YYYY-MM-DD" }`. Parse all titles into a set for dedup matching.

3. **Generate 2x the requested count internally.** Think of `count * 2` ideas.
   For each, check against the dedup set -- skip any idea whose core concept
   overlaps with a prior idea (same mechanism, same insight, same approach even
   if phrased differently). Exact title match is obvious duplication, but also
   filter conceptual duplicates.

4. **Select the best `count` ideas from the surviving pool.** Rank by novelty,
   feasibility (for actionable/technical), and potential impact.

5. **Write the output file.** Determine a topic slug (lowercase, hyphens, no
   special chars, max 60 chars). Path: `projects/ideas/<slug>-YYYY-MM-DD.md`.
   Use the template below.

6. **Append to history.jsonl.** For each idea in the final output, append one
   JSON line to `projects/ideas/history.jsonl`:
   ```json
   {"title": "Idea title", "topic": "original topic", "style": "actionable", "date": "YYYY-MM-DD", "slug": "topic-slug"}
   ```

7. **Commit** to the `projects/ideas/` git repo:
   ```bash
   cd projects/ideas
   git add .
   git commit -m "Ideas: <topic> (<count> ideas, <style>)"
   ```

8. **Update the project CHANGELOG.** Prepend to
   `projects/ideas/.context/CHANGELOG.md`:
   ```
   ## YYYY-MM-DD -- <topic>
   File: <slug>-YYYY-MM-DD.md
   Count: <N> ideas
   Style: <style>
   ```

9. **Final text message.** Your final text block must be a brief list of the
   idea titles (one line each, no descriptions). This becomes the job's summary.

## Output template

```markdown
# Ideas: <Topic>

**Date**: YYYY-MM-DD
**Style**: actionable | speculative | technical-only | product-only
**Count**: N

## <Idea 1 Title>

<2-3 sentence description of the idea.>

**Why now**: <1-2 sentences on why this is timely or newly possible.>

**Hardest part**: <1-2 sentences on the biggest obstacle or risk.>

**First step**: <1 concrete action to start exploring this idea.>

## <Idea 2 Title>

...

## <Idea N Title>

...
```

## Quality gate

Self-check before your final text message:

- [ ] File exists at the correct path with the correct naming pattern
- [ ] Exactly `count` ideas in the output (not more, not less)
- [ ] Each idea has all four sections: description, Why now, Hardest part, First step
- [ ] No idea is a conceptual duplicate of anything in history.jsonl
- [ ] history.jsonl has been appended with one line per idea
- [ ] Style constraint was respected (no product ideas in technical-only, etc.)
- [ ] `projects/ideas/.context/CHANGELOG.md` updated
- [ ] Git commit in `projects/ideas/` exists

## Gotchas (living section -- append when you learn something)

- **Dedup is conceptual, not just string matching**: "Build a CLI dashboard"
  and "Terminal-based monitoring tool" are the same idea. Filter both.
- **Style constraint is strict**: if the user asks for `technical-only`, do not
  include business model or go-to-market ideas even if they're great.
- **Empty history.jsonl is fine**: on first run, there's nothing to dedup against.
- **Count ceiling**: never produce more than 5 ideas per run regardless of what
  the user requests. If they ask for 10, produce 5 and note the cap.
